§ 1.1

What Is an Algorithm? Level 0

▸ The Official Definition

An algorithm is a finite sequence of unambiguous instructions that, given any valid input, always terminates and produces a correct output.

Input  →  [ Algorithm ]  →  Output

Let's Break That Definition Apart

1
Finite sequence There are a countable number of steps. They don't go on forever. Step 1, Step 2, … Step N, done.
must stop
2
Unambiguous instructions Each step has EXACTLY ONE interpretation. "Add some salt" is ambiguous. "Add 5 grams of salt" is unambiguous.
precise
3
Any valid input The algorithm works for all legitimate inputs, not just the ones you tested. If it only works for even numbers but you call it an "any integer" algorithm — it's wrong.
general
4
Always terminates It must stop at some point. A loop that never ends is NOT an algorithm — it's a bug.
finite
5
Produces output Something comes out the other end. An answer, a sorted list, a yes/no — something.
result

🍳 Analogy — A Recipe is an Algorithm

A cooking recipe is almost an algorithm. "Bake at 180°C for 30 minutes" is good — specific and unambiguous. But "bake until golden" is BAD for a computer — it can't look at color. Every instruction must be a concrete, measurable action.

The difference is that a recipe can have vague steps (a human fills in the gaps). An algorithm cannot — a computer has no common sense.

Three Different Algorithms, Same Problem Level 10

The textbook's central point: many algorithms can solve the same problem. Their speeds can differ by factors of millions. Let's see this for the GCD problem.

▸ The GCD Problem

Input: Two positive integers m and n.

Output: The Greatest Common Divisor — the largest integer that divides both m and n exactly (no remainder).

Example: gcd(12, 8) = 4 because 4 divides 12 (12÷4=3 ✓) and 4 divides 8 (8÷4=2 ✓), and no number bigger than 4 does both.

Algorithm 1 — Euclid's Algorithm (2,300 years old! Still the best.)

The mathematical insight: gcd(m, n) = gcd(n, m mod n). Keep replacing the pair until the second number hits zero. The last non-zero number is the answer.

💡 Why Does This Work?

If d divides both m and n, it also divides m−n, m−2n, and so on — basically m mod n. So the GCD of (m, n) equals the GCD of (n, m mod n). We keep reducing the pair until the second number is 0, at which point the first number IS the GCD.

Trust this for now. You'll prove it rigorously in a Number Theory course.

ALGORITHM Euclid(m, n)
// Input:  Two non-negative integers m ≥ n ≥ 0, not both zero
// Output: Greatest common divisor of m and n

while n ≠ 0 do
    r  ← m mod n      // remainder when m is divided by n
    m  ← n             // n becomes the new m
    n  ← r             // remainder becomes the new n
return m               // when n = 0, m is our answer

Verify: 6×8=48 ✓, 6×3=18 ✓. Does 7 divide both? 48÷7 ≈ 6.86 ✗. So 6 is indeed the GCD.

Algorithm 2 — Consecutive Integer Checking (Simple but Slow)

Try every number from min(m,n) down to 1. The first one that divides both is the GCD.

ALGORITHM ConsecutiveCheck(m, n)
// Start at the smaller of m, n and count down
t ← min(m, n)
while t > 0 do
    if (m mod t = 0) AND (n mod t = 0)
        return t      // first t that divides both = GCD
    t ← t - 1
⚠ The Speed Problem

For gcd(60, 24): Euclid takes 2 steps. Consecutive checking starts at t=24 and tries 24, 23, 22, 21, 20, 19, 18, 17, 16, 15, 14, 13, 12 — that's 13 steps.

For gcd(1,000,000, 999,983): Euclid takes about 4 steps. Consecutive checking tries almost a million values. Same answer. Completely different effort.

Algorithm 3 — Middle-School Method (List all divisors, find common ones)

Find all divisors of m. Find all divisors of n. The largest number in both lists is the GCD.

This is how you probably learned it in school. It's correct. It's also the slowest of the three. The textbook mentions this to show that "obvious" is not the same as "efficient."

★ Big Takeaway

Three algorithms. Same correct answer. Speeds that differ by factors of thousands or millions. This is why algorithm design matters. Choosing the right algorithm is not a minor detail — for large inputs, it's the difference between a fast app and one that never finishes.

The Sieve of Eratosthenes — Finding All Primes Level 15

▸ The Problem

Input: A positive integer n.

Output: All prime numbers from 2 to n.

A prime number is a number that has exactly two divisors: 1 and itself. Examples: 2, 3, 5, 7, 11, 13 ...

💡 The Sieve Idea

Write all numbers 2 to n on paper. Start with p=2. Cross out all multiples of 2 (4, 6, 8...) — they can't be prime. Move to the next un-crossed number (3). Cross out all multiples of 3. Keep going until p > √n. What's left, un-crossed, are the primes.

Why stop at √n? Because if a number has a factor larger than √n, it must also have one smaller than √n — and we'd have already crossed it out.

ALGORITHM Sieve(n)
// Input:  Integer n ≥ 2
// Output: Array of all primes up to n

// Step 1: Fill array with 2 to n
for p ← 2 to n do
    A[p] ← p

// Step 2: Cross out multiples
for p ← 2 to ⌊√n⌋ do
    if A[p] ≠ 0             // p not yet crossed out = still prime candidate
        j ← p × p           // start from p² (smaller multiples already crossed)
        while j ≤ n do
            A[j] ← 0        // cross out this multiple
            j ← j + p        // next multiple of p

// Step 3: Collect survivors
return all A[p] where A[p] ≠ 0

§ 1.2

How to Design an Algorithm Level 20

Section 1.2 is about process. How do you go from a blank page to a working algorithm? There is a systematic 7-stage approach. Skipping stages is how bugs are born.

The 7-Stage Algorithm Design Process

1
Understand the Problem. This sounds obvious, but it's where most beginners fail. Read the problem statement twice. Then write down: What are the inputs? What exactly should the output be? What are the edge cases? What if the input is empty? What if all elements are equal?
Never write code before you can state the problem in your own words.
2
Ascertain Your Device's Capabilities. Are you solving this on a supercomputer or a mobile phone? Is the dataset tiny (100 items) or huge (1 billion)? A quadratic algorithm might be fine on a phone for 100 items but disaster for 1 million.
3
Exact or Approximate? For sorting, you want exact. For computing π, you get approximate. For some optimization problems (like the Traveling Salesman), exact is computationally impossible for large inputs — so we accept approximate answers that are "close enough." Knowing which mode you're in changes your entire strategy.
4
Choose a Design Technique. Algorithm design has standard "templates." Brute force, divide-and-conquer, dynamic programming, greedy, backtracking — these are cooking methods (fry vs bake vs steam). The same ingredients cooked differently give completely different results. Chapter 3 starts with brute force — the simplest template.
5
Design the Algorithm and Data Structures. Write pseudocode. Pick data structures. An array gives O(1) access by index. A linked list gives O(n) access. A hash table gives O(1) average lookup. The wrong structure can slow a correct algorithm by 100×.
6
Prove Correctness. Does it always give the right answer? Test every edge case. Can you construct a mathematical proof? One counterexample kills an algorithm. "It worked on 10 test cases" is not a proof.
7
Analyze Efficiency. How fast? How much memory? Big-O notation (Chapter 2) is the tool. A correct but slow algorithm is often useless in practice.
✏ Classic Puzzle — The River Crossing Problem (Stage 1 in action)

A farmer has a wolf, a goat, and a cabbage. He needs to cross a river. His boat holds only him plus one item. Rules: wolf + goat alone = wolf eats goat. Goat + cabbage alone = goat eats cabbage. Wolf + cabbage alone = safe.

The key insight (Stage 1 — Understand): The goat is the problem. It can eat cabbage and can be eaten. It must never be left alone with either. The solution requires going backward — which seems counterintuitive.

1
Take the goat across. (Wolf + cabbage left alone — safe.)
2
Return alone.
3
Take the wolf across.
4
Bring the goat back. ← The non-obvious step! You un-do progress to prevent conflict.
5
Take the cabbage across. (Goat is back on the start side, alone — fine.)
6
Return alone.
7
Take the goat across. ✅ All three are on the far side.

Step 4 only becomes obvious when you formally model the constraint: the goat must not be alone with either. This is Stage 1 doing its job.

Qualities of a Good Algorithm

QualityMeaningTrade-Off
CorrectnessAlways produces the right answer for all valid inputsNon-negotiable — but hard to prove formally
EfficiencyFast runtime and low memory usageOften conflicts with simplicity
SimplicityEasy to read, understand, and verifySimpler code has fewer bugs

★ Design Philosophy

"A designer knows he has arrived at perfection not when there is nothing left to add, but when there is nothing left to take away." — Antoine de Saint-Exupéry

The best algorithms are elegant — they feel inevitable once you see them. Euclid's algorithm is 2,300 years old and still perfect. Simplicity and correctness beat cleverness and complexity every time.

§ 2.1

The Analysis Framework Level 25

How do you know if an algorithm is fast or slow without running it? This section gives you a rigorous framework that works the same regardless of what machine you use.

Why Not Just Time It?

MethodProblem with it
Measure in secondsMy laptop runs it in 0.2s; your laptop in 1.8s; a 2010 computer in 45s. Meaningless comparison.
Count ALL machine instructionsToo complex. Addition, comparison, memory access — hundreds of different micro-operations obscure the pattern.
Count only the basic operation ✓Machine-independent. Captures exactly what scales with input size.
▸ Basic Operation

The basic operation is the single operation that is executed most often and contributes most to the algorithm's total running time. It lives almost always in the innermost loop.

We write C(n) for the count of basic operations on input of size n. Then: T(n) ≈ c · C(n) where c is the (constant, hardware-dependent) time for one basic operation.

AlgorithmBasic OperationWhy?
SortingKey comparison: A[i] < A[j]?Determines order; drives all decisions
SearchingEquality check: A[i] == target?The check that runs in every iteration
Matrix arithmeticMultiplication of two numbersMultiplication costs more than addition
Graph traversalVisiting a vertex or edgeThe per-node/per-edge work

Orders of Growth — The Most Important Concept in This Course Level 35

💡 Why Growth Rate Beats Everything Else

Algorithm A does 1,000,000 basic operations for n=100. Algorithm B does n² = 10,000 for n=100. A wins. But now try n=10,000. A still does 1,000,000 (it's constant!). B does 100,000,000. B is now 100× slower. Flip to n=1,000,000 and B does a trillion operations. The constant factor vanishes. Growth rate is everything at scale.

Worst-Case, Best-Case, Average-Case Level 45

The same algorithm on the same size input can behave completely differently depending on which input it gets. We analyze three scenarios:

CaseWhen It OccursSequential Search ExampleWhy We Care
Best CaseThe most favorable input for this algorithmTarget is the first element → 1 comparisonLower bound on time — algorithm can't do better
Worst Case ★The most difficult input for this algorithmTarget is last or absent → n comparisonsGuarantees performance ceiling — most useful
Average CaseExpected behavior on a random inputTarget at random position → ~n/2 comparisonsReal-world expected performance
★ Why Worst-Case Dominates

When you design a hospital system, you plan for the worst case — the busiest night, not the quietest. When Google Search runs, it cannot afford to occasionally take 10 minutes just because the query was hard. Worst-case is a contractual guarantee. We almost always analyze and report it.

§ 2.2

Asymptotic Notations: O, Ω, and Θ Level 50

We now have the mathematical language to talk about efficiency precisely. Three notations let us classify algorithms by growth rate, ignoring constant factors.

💡 Why Ignore Constants?

Algorithm A does 100n comparisons. Algorithm B does 2n comparisons. A is "50× slower." But for n = 1,000,000: A does 100,000,000 and B does 2,000,000. Both finish in under a second. Now compare algorithm C that does n² = 10¹² comparisons — it takes 1,000 seconds. C is categorically in a different class. The growth-rate category matters enormously; the constant within a category matters much less.

The Efficiency Hierarchy

O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(n³) < O(2ⁿ) < O(n!)

← FASTER / LESS WORK                       SLOWER / MORE WORK →

Big-O — Upper Bound ("at most this much work") Level 52

▸ Formal Definition

t(n) ∈ O(g(n)) if there exist constants c > 0 and n₀ ≥ 0 such that:

t(n) ≤ c · g(n)   for all n ≥ n₀

Plain English: "Beyond some input size n₀, t(n) never exceeds c · g(n)." It's an upper bound — a ceiling on growth.

Big-O: t(n) stays at or below c·g(n) for all n ≥ n₀
✏ Proof — Show that 100n + 5 ∈ O(n²)

Goal: Find c > 0 and n₀ ≥ 0 such that 100n + 5 ≤ c·n² for all n ≥ n₀.

Strategy: Overestimate the left side first.

For n ≥ 5:
100n + 5 ≤ 100n + n = 101n        (since 5 ≤ n when n ≥ 5)
101n ≤ 101·n²                      (since n ≤ n² when n ≥ 1)
∴ 100n + 5 ≤ 101·n²

Choose c = 101, n₀ = 5. ✓ Proof complete.

Note: 100n+5 is actually linear (Θ(n)). Big-O is an upper bound — a linear function also fits inside O(n²). But it's a loose bound. We prefer tight bounds (Θ) when possible.

Big-Ω — Lower Bound ("at least this much work") Level 55

▸ Formal Definition

t(n) ∈ Ω(g(n)) if there exist constants c > 0 and n₀ ≥ 0 such that:

t(n) ≥ c · g(n)   for all n ≥ n₀

Plain English: "Beyond n₀, t(n) is always at least c·g(n)." It's a lower bound — a floor on growth.

✏ Proof — Show that n³ ∈ Ω(n²)

Goal: Find c > 0 and n₀ such that n³ ≥ c·n² for all n ≥ n₀.

n³ / n² = n ≥ 1 for all n ≥ 1
∴ n³ ≥ 1·n² = n²

Choose c = 1, n₀ = 1. ✓ n³ grows at least as fast as n².

Big-Θ — Tight Bound ("exactly this growth rate") Level 60

▸ Formal Definition

t(n) ∈ Θ(g(n)) if there exist constants c₁, c₂ > 0 and n₀ ≥ 0 such that:

c₂·g(n) ≤ t(n) ≤ c₁·g(n)   for all n ≥ n₀

Plain English: "t(n) is sandwiched between two multiples of g(n)." It grows exactly like g(n). This is the most useful and precise notation.

✏ Proof — Show that ½n(n−1) ∈ Θ(n²)

Note: ½n(n−1) = ½n² − ½n. This is the number of comparisons in Selection Sort.

Upper bound (show ≤ c₁·n²):

½n² − ½n ≤ ½n²    for all n ≥ 0   [since we're subtracting ½n]

Lower bound (show ≥ c₂·n²):

½n² − ½n ≥ ¼n²   for n ≥ 2
½n² − ¼n² = ¼n² ≤ ½n² − ½n ?   when ½n ≤ ¼n², i.e., n ≥ 2 ✓

Result: ¼n² ≤ ½n(n−1) ≤ ½n² for all n ≥ 2.

Choose c₁ = ½, c₂ = ¼, n₀ = 2. ∴ ½n(n−1) ∈ Θ(n²) ✓

The Limit Method — Easier Than Formal Proofs Level 65

Instead of hunting for constants, compute the limit of the ratio t(n)/g(n) as n → ∞:

lim (t(n)/g(n)) = ?ConclusionNotation
0t grows slower than gt ∈ O(g) only (not Θ)
c (constant > 0)t and g grow at the same ratet ∈ Θ(g) ← most precise
∞t grows faster than gt ∈ Ω(g) only (not Θ)
✏ Compare log₂n vs √n using limits

We want to know: which grows faster as n→∞?

lim(n→∞)   log₂n / √n
Apply L'Hôpital's Rule:   = lim (1/(n·ln2)) / (1/(2√n))
                          = lim (2√n) / (n·ln2)
                          = lim 2/(√n · ln2) → 0

Limit = 0 → log₂n grows slower than √n. So log₂n ∈ O(√n) ✓

§ 2.3

Analyzing Non-Recursive Algorithms Level 65

A non-recursive algorithm uses only loops — no self-calls. Analysis means writing a summation for the basic operation count and simplifying it to a Θ expression.

▸ The 5-Step Recipe for Non-Recursive Analysis
1
Decide on the input-size parameter n.
2
Identify the basic operation — it's almost always in the innermost loop.
3
Check if the count varies for different inputs of the same size. If yes, analyze worst/avg/best separately.
4
Write the sum C(n) = Σ (number of times basic op executes). Use loop bounds to set up the sum.
5
Simplify using summation formulas. Express as Θ(...).

Key Summation Formulas (Memorize These)

// Sum of 1s from l to u (counting iterations):
∑(i = l to u)  1    =  u − l + 1

// Sum of integers 1 to n:
∑(i = 1 to n)  i    =  n(n+1)/2  ≈  n²/2

// Sum of squares (less common, good to know):
∑(i = 1 to n)  i²   =  n(n+1)(2n+1)/6  ≈  n³/3

// Geometric series:
∑(i = 0 to n)  aⁱ   =  (aⁿ⁺¹ − 1) / (a − 1)

Example 1 — Finding Maximum Element (One Loop)

ALGORITHM MaxElement(A[0..n−1])
// Input:  Array A of n real numbers
// Output: Maximum value in A
maxval ← A[0]
for i ← 1 to n−1 do
    if A[i] > maxval      // ← BASIC OPERATION: comparison
        maxval ← A[i]
return maxval

📐 Full Analysis

n = number of elements in A.

Basic operation = A[i] > maxval (the comparison on every iteration).

Does count vary per input? No — the loop always runs from i=1 to n−1, regardless of array values.

C(n) = ∑(i=1 to n−1) 1 = (n−1) − 1 + 1 = n − 1

Result: C(n) = n−1 ∈ Θ(n)

The algorithm always makes exactly n−1 comparisons. Best case = worst case = Θ(n).

Example 2 — Checking All Elements Unique (Two Nested Loops) Level 68

ALGORITHM UniqueElements(A[0..n−1])
// Output: true if all elements are distinct, false otherwise
for i ← 0 to n−2 do
    for j ← i+1 to n−1 do
        if A[i] = A[j]      // ← BASIC OPERATION
            return false    // found a duplicate!
return true             // no duplicates found
💡 What Pairs Are Being Compared?

For n=4: (A[0],A[1]), (A[0],A[2]), (A[0],A[3]), (A[1],A[2]), (A[1],A[3]), (A[2],A[3]) — 6 pairs = 4×3/2. For n=5: 10 pairs = 5×4/2. This is the upper triangle of the comparison matrix.

📐 Full Analysis

Basic operation = A[i] = A[j]

Does count vary? YES — if duplicates are found early, the loop exits early. So we analyze worst case.

Worst case: All elements are unique — we compare all pairs.

C_worst(n) = ∑(i=0 to n−2) ∑(j=i+1 to n−1) 1
           = ∑(i=0 to n−2) (n−1−i)
           = (n−1) + (n−2) + (n−3) + ... + 1
           = n(n−1)/2

Result: C_worst(n) = n(n−1)/2 ∈ Θ(n²)

Two nested loops where the inner loop depends on i → quadratic. This is the classic signature.

Example 3 — Matrix Multiplication (Three Nested Loops) Level 72

ALGORITHM MatrixMult(A[0..n−1, 0..n−1], B[0..n−1, 0..n−1])
// Output: Matrix C = A × B (n×n result matrix)
for i ← 0 to n−1 do
    for j ← 0 to n−1 do
        C[i,j] ← 0
        for k ← 0 to n−1 do
            C[i,j] ← C[i,j] + A[i,k] × B[k,j]   // ← BASIC OP
📐 Analysis

Basic operation = Multiplication A[i,k] × B[k,j]

C(n) = ∑(i=0 to n−1) ∑(j=0 to n−1) ∑(k=0 to n−1) 1 = n · n · n = n³

Result: C(n) = n³ ∈ Θ(n³)

For a 1000×1000 matrix: 10⁹ multiplications. For 10,000×10,000: 10¹² — at 10⁹ ops/sec that's 1,000 seconds. This is why advanced matrix algorithms (Strassen: O(n^2.81)) matter in practice.

★ The Loop Counting Rule

1 loop of size n → C(n) grows like n → Θ(n)

2 nested loops of size n → C(n) grows like n² → Θ(n²)

3 nested loops of size n → C(n) grows like n³ → Θ(n³)

This is a heuristic — it works when each loop runs O(n) times independently. Always verify with the summation.

§ 2.4

Analyzing Recursive Algorithms Level 75

A recursive algorithm solves a problem by calling itself on smaller inputs. We can't use a summation — instead we write a recurrence relation and solve it using backward substitution.

▸ Recurrence Relation

An equation where M(n) is expressed in terms of M(smaller values). For example: M(n) = M(n−1) + 1.

To solve it means to find a closed-form expression — M(n) in terms of only n, with no M on the right side.

▸ The 5-Step Recipe for Recursive Analysis
1
Choose input-size parameter n.
2
Identify the basic operation.
3
Determine if count varies per input.
4
Write the recurrence relation with initial condition (base case).
5
Solve by backward substitution: substitute the recurrence into itself repeatedly, find the pattern, plug in the base case.

Example 1 — Factorial n! (Simplest Recursion)

ALGORITHM F(n)
// Computes n! recursively
// F(0) = 1, F(n) = n × F(n−1)
if n = 0
    return 1
else
    return F(n−1) × n     // ONE multiplication here
📐 Setting Up the Recurrence

Basic operation: Multiplication (F(n−1) × n).

Each call does 1 multiplication, plus whatever F(n−1) does.

M(n) = M(n−1) + 1     ← recurrence
M(0) = 0                   ← base case (no multiplication needed)

📐 Solving by Backward Substitution

M(n) = M(n−1) + 1
     = [M(n−2) + 1] + 1             = M(n−2) + 2
     = [M(n−3) + 1] + 2             = M(n−3) + 3
     ⋮
     = M(n−k) + k            ← pattern after k substitutions

Set k = n: M(n) = M(n−n) + n = M(0) + n = 0 + n = n

✅ M(n) = n ∈ Θ(n) — factorial is computed in linear time.

Example 2 — Tower of Hanoi (The Famous Exponential) Level 80

🗼 The Problem Setup

n disks of different sizes on peg A. Goal: move all to peg C using peg B as helper. Rules: (1) move one disk at a time. (2) never place a larger disk on a smaller one.

The key insight: to move the bottom (largest) disk, you must first move all n−1 others out of the way. This creates two recursive subproblems of size n−1.

ALGORITHM Hanoi(n, from, to, via)
if n = 1
    print "move disk 1 from" from "to" to   // 1 move
else
    Hanoi(n−1, from, via, to)   // move n−1 disks out of the way
    print "move disk n from" from "to" to   // 1 move
    Hanoi(n−1, via, to, from)   // move n−1 disks onto destination
Hanoi(3) Recursion Tree — 2 sub-calls per node → tree doubles every level → 2ⁿ−1 = 7 total moves
📐 Recurrence and Solution

Basic operation: Moving one disk.

M(n) = 2·M(n−1) + 1    (two sub-problems + 1 direct move)
M(1) = 1                      (base case: 1 disk needs 1 move)

Backward substitution:

M(n) = 2·M(n−1) + 1
     = 2·[2M(n−2) + 1] + 1        = 4M(n−2) + 3
     = 4·[2M(n−3) + 1] + 3        = 8M(n−3) + 7
     = 2ᵏ·M(n−k) + (2ᵏ − 1)     ← pattern
Set k = n−1:
M(n) = 2ⁿ⁻¹·M(1) + (2ⁿ⁻¹ − 1)
     = 2ⁿ⁻¹ · 1 + 2ⁿ⁻¹ − 1
     = 2ⁿ − 1

✅ M(n) = 2ⁿ − 1 ∈ Θ(2ⁿ) — EXPONENTIAL

For n = 64 disks: 2⁶⁴ − 1 ≈ 1.84 × 10¹⁹ moves. At one move per second: 585 billion years. The universe is ~14 billion years old. This is what exponential growth truly means.

Example 3 — Counting Binary Digits (log Recursion) Level 82

ALGORITHM BinRec(n)
// Counts the number of bits in n's binary representation
// e.g. 13 = 1101₂ has 4 bits
if n = 1
    return 1
else
    return BinRec(⌊n/2⌋) + 1   // halve n each call
📐 Analysis

Recurrence: A(n) = A(⌊n/2⌋) + 1,   A(1) = 0

For exact analysis, let n = 2ᵏ:

A(2ᵏ) = A(2ᵏ⁻¹) + 1
      = A(2ᵏ⁻²) + 2
      = A(2⁰) + k = A(1) + k = 0 + k = k
Since k = log₂n: A(n) = log₂n

✅ A(n) = ⌊log₂n⌋ ∈ Θ(log n) — halving the input → logarithmic growth.

★ Recursion Pattern Cheat Sheet

Call itself ONCE on (n−1): M(n) = M(n−1) + c → Θ(n)

Call itself ONCE on (n/2): M(n) = M(n/2) + c → Θ(log n)   ← binary search, bit counting

Call itself TWICE on (n/2): M(n) = 2M(n/2) + c → Θ(n)   ← full tree has ~n leaves

Call itself TWICE on (n−1): M(n) = 2M(n−1) + c → Θ(2ⁿ)   ← Hanoi, tree doubles every level

Chapter 3

Brute Force & Exhaustive Search

The simplest design strategy — try everything, pick what works. Always correct. Not always fast.

Level 35–90
💡 What is Brute Force?

Brute force is the strategy of solving a problem by trying all possibilities in the most direct way, without cleverness or optimization. Like opening a 4-digit combination lock by trying 0000, 0001, 0002 ... 9999 — it always works, but there are 10,000 tries in the worst case.

Why study brute force? Because it's always the starting point. It sets the baseline. It proves the problem is solvable. Then we improve on it.

§ 3.1

Selection Sort & Bubble Sort Level 55

Selection Sort — "Find the minimum, drag it to the front"

💡 The Idea

On pass i (from 0 to n−2): Scan the entire unsorted portion (A[i] to A[n−1]). Find the minimum element. Swap it into position i. After pass i, the first i+1 elements are sorted and in their final positions forever.

ALGORITHM SelectionSort(A[0..n−1])
// Input:  Array A of n orderable elements
// Output: Array A sorted in ascending order
for i ← 0 to n−2 do
    min_idx ← i                 // assume current pos is minimum
    for j ← i+1 to n−1 do
        if A[j] < A[min_idx]    // ← BASIC OPERATION: comparison
            min_idx ← j         // found a new minimum
    swap(A[i], A[min_idx])      // put minimum in correct place

📐 Complete Analysis

Basic operation: Comparison A[j] < A[min_idx]

Does count vary? No — both loops always run to their bounds regardless of array values.

C(n) = ∑(i=0 to n−2) ∑(j=i+1 to n−1) 1
     = ∑(i=0 to n−2) (n−1 − i)
     = (n−1) + (n−2) + ... + 1
     = n(n−1)/2

✅ C(n) = n(n−1)/2 ∈ Θ(n²) — worst = average = best case

Selection Sort always makes exactly n(n−1)/2 comparisons, regardless of whether the array is sorted or not. That's both a weakness (can't stop early) and a strength (predictable).

Swaps: Always exactly n−1 swaps (one per outer loop pass). This minimizes data movement — useful when swaps are expensive.

Bubble Sort — "Repeatedly compare neighbors, large values rise to the right"

💡 The Idea

Repeatedly scan left-to-right, comparing each adjacent pair A[j] and A[j+1]. Swap if A[j] > A[j+1]. After each full pass, the largest un-placed element has "bubbled" to its final position at the right end. Like bubbles rising in water — heavy things sink, light things rise.

ALGORITHM BubbleSort(A[0..n−1])
for i ← 0 to n−2 do
    for j ← 0 to n−2−i do    // inner loop shrinks each pass
        if A[j+1] < A[j]          // ← BASIC OPERATION
            swap(A[j], A[j+1])  // swap if out of order

Sequential Search & String Matching Level 60

Sequential Search (Linear Search)

The simplest search: scan from left to right until you find the target or reach the end.

ALGORITHM SequentialSearch(A[0..n−1], K)
// Input:  Array A of n elements, search key K
// Output: Index of K in A, or −1 if not found

A[n] ← K          // Place sentinel — guarantees loop WILL terminate
i ← 0
while A[i] ≠ K do
    i ← i + 1
if i < n
    return i       // found at index i
else
    return −1     // not found (hit the sentinel)
💡 The Sentinel Trick

Without the sentinel, the while loop needs TWO conditions each iteration: (1) is i < n? (2) is A[i] = K? With the sentinel (placing K at position n), condition (1) can be removed — the loop will always find K somewhere. Then we just check if the found position is real or the sentinel. This cuts the loop's work in half in some implementations.

📐 Analysis

Best case: K is A[0] → 1 comparison. C_best(n) = 1 ∈ O(1)

Worst case: K is A[n−1] or not in A → n comparisons. C_worst(n) = n ∈ Θ(n)

Average case: Assuming K is present with probability p, at a uniformly random position:

C_avg ≈ p·(n+1)/2 + (1−p)·n

If p=1 (K always present): C_avg = (n+1)/2 ∈ Θ(n)

So best and worst differ only by a factor of n — same asymptotic class. Linear is the best we can do without additional structure.

Brute-Force String Matching Level 65

▸ The Problem

Input: Text T[0..n−1] of length n. Pattern P[0..m−1] of length m. (m ≤ n)

Output: Index of the leftmost occurrence of P in T, or −1 if not found.

💡 The Brute Force Idea

Try every possible alignment. Slide the pattern over the text one position at a time. At each position i, check if P matches T[i..i+m−1] character by character. If all m characters match — found it! If any mismatch — slide one position right and try again.

ALGORITHM BruteForceStringMatch(T[0..n−1], P[0..m−1])
// Input:  Text T (length n), Pattern P (length m)
// Output: Index i where P first appears in T, or −1

for i ← 0 to n−m do       // try each starting position in T
    j ← 0
    while j < m AND T[i+j] = P[j] do
        j ← j + 1             // characters match, check next
    if j = m                   // survived all m checks = full match
        return i
return −1                   // no match found
✏ Detailed Trace — T = "ABCABCABD", P = "ABCABD"

i=0: A=A✓ B=B✓ C=C✓ A=A✓ B=B✓ D≠C✗ → mismatch at j=5

i=1: A≠B✗ → mismatch at j=0

i=2: A≠C✗ → mismatch at j=0

i=3: A=A✓ B=B✓ C=C✓ A=A✓ B=B✓ D=D✓ → j=6=m! Found at index 3! ✅

✏ Worst-Case Input — T = "AAAAAAB", P = "AAAB"

At each position: AAA match, then last character mismatches. We do m comparisons before each rejection.

Number of positions: n−m+1 = 7−4+1 = 4

Total comparisons: 4 × 4 = 16 = m × (n−m+1)

This is the worst-case scenario for string matching.

📐 Analysis

Basic operation: Character comparison T[i+j] = P[j]

Best case: Match found at position 0 after m comparisons. C_best = m ∈ O(m)

Worst case: (n−m+1) positions × m comparisons each = m(n−m+1). For m ≪ n: ≈ mn. C_worst ∈ O(mn)

Average case: In practice, most mismatches happen after 1–2 comparisons (natural language texts). C_avg ≈ O(n) in practice.

★ Beyond Brute Force String Matching

The brute-force O(mn) algorithm is actually used in many real situations (small patterns, short texts). But for large-scale search (e.g., searching a genome database), smarter algorithms exist:

• KMP (Knuth-Morris-Pratt): O(n+m) always — never re-examines text characters

• Boyer-Moore: O(n/m) best case — skips whole sections of text

• Rabin-Karp: Uses hashing for fast comparison — O(n+m) average

These are future topics. Brute force is the foundation.

★ Final Wisdom

You now know the foundations of algorithm analysis. The questions to always ask are: (1) What grows with input size? (find the basic operation) (2) How does it grow? (set up the sum or recurrence) (3) What class is it in? (simplify to Θ notation) (4) Is that acceptable? (compare against the table above).

Every advanced algorithm in the field — QuickSort, Dijkstra, A*, FFT, dynamic programming — is analyzed using exactly these tools. You now have the language to read and understand them.
